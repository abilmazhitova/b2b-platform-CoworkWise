## 1. ВВЕДЕНИЕ

### 1.1 Актуальность темы

Рынок коворкингов в Казахстане демонстрирует устойчивый рост: по данным аналитических агентств, с 2019 по 2024 год количество коворкинг-пространств в Алматы увеличилось более чем втрое. Расширение удалённого и гибридного формата работы, рост числа фрилансеров и малого бизнеса формируют стабильный спрос на гибкие офисные пространства.

Вместе с тем выбор локации для открытия коворкинга остаётся одним из наиболее рискованных решений для инвестора: по оценкам, до 40% новых коворкингов закрываются в течение первых двух лет, и основной причиной служит неверная оценка потенциала местоположения. Традиционный подход — субъективный осмотр района, визуальная оценка пешеходного трафика и сравнение нескольких вариантов «на глаз» — не позволяет обработать многомерные данные и принять объективное решение.

Актуальность разработки аналитической платформы, объединяющей реальные данные о перемещениях людей, геопространственную инфраструктуру и рыночные ставки аренды в единый инструмент поддержки принятия решений, не вызывает сомнений.

### 1.2 Цель и задачи работы

**Цель:** разработать веб-платформу CoworkWise для B2B-аудитории (девелоперы, инвесторы, операторы коворкингов), которая на основе телекоммуникационных данных и геопространственного анализа автоматически ранжирует локации Алматы по потенциалу открытия коворкинга.

**Задачи:**
1. Исследовать предметную область и выявить ключевые факторы успешности коворкинга.
2. Разработать алгоритм многофакторного скоринга локаций.
3. Спроектировать и реализовать серверную часть (REST API) с хранением телеком-данных в реляционной СУБД.
4. Реализовать клиентскую часть с интерактивной картой, аналитическими инструментами и трёхязычным интерфейсом.
5. Интегрировать данные из открытых источников: OpenStreetMap (инфраструктура), Krisha.kz (коммерческая аренда).
6. Обеспечить развёртывание системы в Docker-контейнерах.

### 1.3 Практическая значимость

Платформа позволяет:
- сократить время первичного анализа локаций с нескольких недель до нескольких минут;
- сравнивать произвольный набор точек по единой системе метрик;
- получить обоснованный рейтинг на основе реальных данных о посещаемости, а не экспертных оценок.

## 2. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

### 2.1 Факторы успешности коворкинга

На основе анализа литературы и реальных кейсов рынка Алматы были выделены пять ключевых факторов, определяющих потенциал локации:

| Фактор | Влияние на успех | Источник данных |
|--------|-----------------|-----------------|
| Пешеходный трафик | Основной — высокая посещаемость района означает органический поток клиентов | Данные телеком-оператора |
| Транспортная доступность | Высокое — близость метро и автобусных остановок расширяет охват аудитории | OpenStreetMap |
| Деловая инфраструктура | Среднее — кафе, университеты, бизнес-центры создают «экосистему» | OpenStreetMap |
| Конкуренция | Отрицательное — высокая насыщенность рынка снижает долю нового игрока | OpenStreetMap |
| Стоимость аренды | Отрицательное — напрямую влияет на ROI и срок окупаемости | Krisha.kz (парсинг) |

### 2.2 Обзор существующих решений

**CBRE / JLL Analytics** — коммерческие платформы для анализа коммерческой недвижимости; не адаптированы для казахстанского рынка, высокая стоимость доступа.

**Google Maps Business Intelligence** — общий инструмент, не предоставляет данных о реальном трафике по ячейкам и не имеет алгоритмов оценки потенциала.

**2GIS Pro** — покрывает Алматы, но API предоставляет лишь статические данные о POI, не включает телеком-сигнал.

**CoworkWise** занимает незаполненную нишу: казахстанская платформа с реальными телеком-данными, открытой геоинфраструктурой и локальными данными об аренде.

### 2.3 Источники данных

**Телекоммуникационные данные.** Оператор предоставляет агрегированные данные в формате `.xlsx`: количество уникальных пользователей в каждой географической ячейке (ZID) с разбивкой по дням недели и часам суток. Данные анонимны и не содержат персональных сведений.

**OpenStreetMap (Overpass API).** Открытая картографическая база данных, содержащая объекты инфраструктуры: остановки общественного транспорта, кафе, рестораны, университеты, торговые центры, спортивные объекты, существующие коворкинги.

**Krisha.kz.** Крупнейший казахстанский агрегатор объявлений о недвижимости. Данные о коммерческой аренде собраны методом парсинга: 455 объявлений с ценой аренды тг/м²/мес по 8 районам Алматы.

## 3. ПОСТАНОВКА ЗАДАЧИ И ТРЕБОВАНИЯ К СИСТЕМЕ

### 3.1 Функциональные требования

**FR-01 Аутентификация:** система должна обеспечивать регистрацию и авторизацию пользователей с хранением JWT-токена.

**FR-02 Загрузка данных:** администратор должен иметь возможность загружать `.xlsx`-файлы с телеком-данными через веб-интерфейс с указанием метки месяца.

**FR-03 Карта трафика:** система должна отображать интерактивную тепловую карту пешеходного трафика по ячейкам Алматы с фильтрацией по районам, времени суток и радиусу поиска.

**FR-04 Рекомендации:** система должна автоматически формировать список топ-15 локаций с оценкой потенциала (0–100), текстовыми преимуществами и метриками.

**FR-05 Сравнение:** пользователь должен иметь возможность выбрать несколько точек на карте и сравнить их по единой таблице с показателями трафика, конкуренции, аренды и итогового рейтинга.

**FR-06 Прогноз:** система должна отображать прогноз трафика по месяцам и рейтинг районов по темпу роста.

**FR-07 Панель администратора:** управление пользователями, назначение ролей, настройки системы.

**FR-08 Мультиязычность:** интерфейс должен поддерживать три языка — английский, русский, казахский.

### 3.2 Нефункциональные требования

| Требование | Значение |
|-----------|----------|
| Доступность | Веб-приложение, доступное через браузер без установки ПО |
| Производительность | Время ответа API — не более 5 секунд для аналитических запросов |
| Безопасность | Хэширование паролей (bcrypt), JWT, разграничение ролей admin/user |
| Масштабируемость | Контейнеризация Docker, независимое масштабирование сервисов |
| Переносимость | Запуск на любой ОС с поддержкой Docker |

### 3.3 Ограничения

- ML-аналитический модуль реализует алгоритм экспертного скоринга с использованием инструментов машинного обучения (scikit-learn). Веса факторов определены на основе анализа предметной области. Архитектура модуля спроектирована так, что при накоплении размеченных исторических данных фиксированные веса могут быть заменены обученной моделью без изменения остальной системы.
- Данные об аренде представляют собой медианные «запрашиваемые» цены, а не фактические сделки.
- Система охватывает 8 административных районов Алматы.

## 4. ВЫБОР И ОБОСНОВАНИЕ ТЕХНОЛОГИЙ

### 4.1 Серверная часть

**Python 3.12 + FastAPI** выбраны как серверная платформа по следующим причинам:
- FastAPI обеспечивает автоматическую генерацию OpenAPI-документации и нативную поддержку асинхронных операций через `asyncio`;
- богатая экосистема Python для геопространственного анализа (GeoPandas, Shapely, scikit-learn) критически важна для ML-сервиса;
- высокая производительность (сопоставима с Node.js) при асинхронных I/O-операциях.

**SQLAlchemy 2.0 (async) + asyncpg** — ORM с поддержкой асинхронных запросов к PostgreSQL, что позволяет обрабатывать большое количество одновременных запросов без блокировки.

**PostgreSQL 16** — реляционная СУБД, обеспечивающая надёжное хранение телеком-данных (потенциально миллионы строк в `telecom_stats`), транзакционность и возможность будущего расширения с PostGIS.

**Alembic** — система управления миграциями схемы базы данных, обеспечивающая версионирование и воспроизводимое развёртывание.

### 4.2 Клиентская часть

**Next.js 15 (App Router) + TypeScript** — фреймворк для React с серверным рендерингом, файловой маршрутизацией и встроенной оптимизацией. TypeScript обеспечивает типобезопасность и снижает количество ошибок при разработке.

**Tailwind CSS v4 + shadcn/ui** — утилитарная CSS-система и набор доступных UI-компонентов на базе Radix UI; обеспечивают единый дизайн без необходимости писать кастомный CSS.

**Leaflet** — библиотека для интерактивных карт с поддержкой тепловых слоёв (leaflet.heat), GeoJSON-полигонов и кастомных маркеров. Выбрана как лёгкая альтернатива Google Maps (бесплатная, без ограничений по ключу API).

**Recharts** — декларативная библиотека графиков для React; использована для гистограмм трафика и линейных графиков прогноза.

**Zustand** — минималистичная библиотека управления глобальным состоянием; хранит фильтры карты, сохранённые точки и язык интерфейса с персистентностью в localStorage.

### 4.3 ML-аналитический модуль

ML-слой платформы реализует полный аналитический пайплайн, следующий классической схеме машинного обучения: сбор данных → извлечение признаков (feature engineering) → нормализация → скоринг → ранжирование. Веса факторов определены экспертно на основе анализа предметной области; архитектура модуля допускает замену на обученную модель при появлении размеченных данных.

**scikit-learn (MinMaxScaler)** — инструмент из стандартной ML-библиотеки; применяется для нормализации признаков в диапазон [0, 1] по формуле (x − min) / (max − min), что обеспечивает сопоставимость разнородных показателей перед скорингом.

**GeoPandas + Shapely** — геопространственные библиотеки для извлечения признаков (feature extraction):
- привязка телеком-ячеек к районам через пространственное объединение (`sjoin`) с полигонами `districts.geojson`;
- подсчёт объектов инфраструктуры в радиусе 700 метров от каждой ячейки — формирование числовых признаков `infra_score`, `accessibility`, `competition`.

### 4.4 Инфраструктура

**Docker + Docker Compose** — контейнеризация трёх независимых сервисов: frontend (Node.js 20), backend (Python 3.12), database (PostgreSQL 16). Обеспечивает воспроизводимое развёртывание и изоляцию зависимостей.

## 5. ПРОЕКТИРОВАНИЕ СИСТЕМЫ

### 5.1 Архитектура

Система построена по трёхзвенной архитектуре клиент–сервер–база данных, развёрнутой в Docker Compose:

```
Браузер пользователя
       │  HTTP :3000
       ▼
┌─────────────────┐
│  Next.js 15     │  b2b-frontend  (порт 3000)
│  (App Router)   │
└────────┬────────┘
         │  REST API  HTTP :8000
         ▼
┌─────────────────┐
│  FastAPI        │  b2b-backend   (порт 8000)
│  + ML-сервис    │
└────────┬────────┘
         │  SQL (asyncpg)
         ▼
┌─────────────────┐
│  PostgreSQL 16  │  b2b-postgres  (порт 5432)
└─────────────────┘
```

Фронтенд взаимодействует с бэкендом исключительно через REST API. Все аналитические вычисления выполняются на сервере.

### 5.2 Схема базы данных

```
┌──────────────────────┐         ┌───────────────────────────────┐
│       users          │         │        telecom_grids           │
├──────────────────────┤         ├───────────────────────────────┤
│ id          PK INT   │         │ id           PK INT            │
│ email       UNIQUE   │         │ zid_number   BIGINT UNIQUE     │
│ hashed_pw   STR      │         │ lat_bot_left FLOAT             │
│ full_name   STR?     │         │ lon_bot_left FLOAT             │
│ is_active   BOOL     │         │ lat_bot_right FLOAT            │
│ is_admin    BOOL     │         │ lon_bot_right FLOAT            │
└──────────────────────┘         │ lat_top_right FLOAT            │
                                 │ lon_top_right FLOAT            │
                                 └──────────────┬────────────────┘
                                                │ 1:N
                                 ┌──────────────▼────────────────┐
                                 │        telecom_stats           │
                                 ├───────────────────────────────┤
                                 │ id          PK INT             │
                                 │ grid_id     FK → grids.id      │
                                 │ week_day    INT (0–6)          │
                                 │ time_hour   INT (0–23)         │

```

Таблица `telecom_grids` хранит географические координаты четырёх угловых точек каждой ячейки. Таблица `telecom_stats` содержит статистику уникальных пользователей с разбивкой по часу и дню недели для каждой загруженной партии данных.

### 5.4 ML-пайплайн: алгоритм многофакторного скоринга

ML-аналитический модуль выполняется в `ml_analysis_service.py`. Пайплайн следует стандартной схеме: сбор данных → feature extraction → нормализация → взвешенный скоринг → ранжирование. Состоит из семи шагов:

**Шаг 1 — Сбор данных из БД.**  
Для каждой ячейки запрашивается суммарное количество уникальных пользователей из `telecom_stats` с учётом активных фильтров времени и дня.

**Шаг 2 — Геопривязка.**  
Центр каждой ячейки ((lat_bot + lat_top)/2, (lon_bot + lon_top)/2) преобразуется в точку GeoDataFrame. Через пространственное объединение (`sjoin`) с полигонами `districts.geojson` определяется принадлежность к одному из 8 районов.

**Шаг 3 — Извлечение геопространственных признаков (Feature Extraction).**  
В системе координат EPSG:3857 (метрическая проекция) для каждой ячейки в радиусе 700 м подсчитываются объекты из `infra_points.csv` — формируются три числовых признака:
- `infra_score` — кафе, рестораны, офисы, университеты, торговые центры, спортивные объекты;
- `accessibility` — остановки автобусов и станции метро;
- `competition` — существующие коворкинги.

**Шаг 4 — Стоимость аренды.**  
По определённому на шаге 2 названию района из справочника `_RENT_INDEX` извлекается медиана коммерческой аренды (тг/м²/мес). При отсутствии данных используется среднее по всем районам.

**Шаг 5 — Нормализация признаков.**  
Четыре количественных признака (трафик, инфра, транспорт, конкуренция) нормализуются через `MinMaxScaler` из scikit-learn по всей выборке ячеек. Аренда нормализуется по фиксированному глобальному диапазону [4 615; 12 100] тг/м². Нормализация приводит все признаки к единому масштабу [0, 1], что является обязательным шагом перед применением взвешенной линейной модели.

**Шаг 6 — Взвешенная линейная модель скоринга.**  
```
potential_score =
    0.30 × трафик_норм
  + 0.23 × инфраструктура_норм
  + 0.17 × транспорт_норм
  − 0.15 × конкуренция_норм
  − 0.15 × аренда_норм
```
По структуре это взвешенная линейная модель — стандартный класс моделей в машинном обучении. Веса определены экспертно на основе анализа предметной области (в отличие от обученных моделей, где веса подбираются из данных). Балл от 0 до 1 преобразуется в шкалу 0–100. Оценки: ≥80 — Excellent, ≥60 — Very Good, остальные — Good.

**Шаг 7 — Текстовые преимущества.**  
По пороговым значениям нормализованных метрик автоматически формируются текстовые метки: «High people density», «Affordable rent», «High competition area» и т.д.

### 5.5 Пользовательский интерфейс (основные экраны)

**Карта (`/map`)** — тепловая карта трафика на базе Leaflet с панелью фильтров (район, радиус, время суток). Режим Explore: клик на точку → аналитика. Режим Compare: выбор 2–5 точек для сравнения.

**Анализ (`/analysis`)** — четыре вкладки: Сравнение, Рекомендации, Сохранённые места, Прогноз.

**Администратор (`/admin`)** — управление пользователями, загрузка телеком-данных, системные настройки.

### 6.2 Обработка телеком-данных

При загрузке `.xlsx`-файла через `POST /telecom/upload` сервис `telecom_service.py` выполняет следующие операции:
1. Читает файл через `openpyxl`.
2. Для каждой строки проверяет уникальность `ZID_NUMBER` — создаёт запись в `telecom_grids` при первом появлении (upsert по `zid_number`).
3. Добавляет запись в `telecom_stats` с привязкой к ячейке, временными атрибутами и меткой месяца.
4. Все операции выполняются в одной асинхронной транзакции для обеспечения целостности данных.

### 6.3 Сбор данных об аренде (парсер Krisha.kz)

Скрипт `scraper/krisha_scraper.py` собирает объявления о коммерческой аренде в Алматы:
- Навигация по страницам листинга через URL-параметр `page`.
- Извлечение встроенного JSON из объекта `window.data` на странице.
- Парсинг цены по регулярному выражению `([\d\xa0\s]+)\s*[〒₸]\s*за\s*м²` с учётом неразрывных пробелов (`\xa0`), используемых Krisha как разделитель тысяч.
- Определение района по строке адреса объявления.
- Фильтрация складской и промышленной недвижимости.
- Агрегация: медиана по каждому из 8 районов.

Результат (455 объявлений, май 2026) сохранён в `b2b-backend/data/rent_index.json`:

### 6.4 Интернационализация (i18n)

Три языка реализованы через клиентскую библиотеку Zustand без серверных зависимостей:
- Словари хранятся в файлах `src/lib/translations/en.ts`, `ru.ts`, `kk.ts`.
- Тип `Translations` выводится из английского файла — TypeScript выдаёт ошибку при отсутствии ключа в русском или казахском словаре.
- Хук `useT()` возвращает нужный словарь по текущему языку из `useLangStore`.
- Выбранный язык сохраняется в localStorage и восстанавливается при следующей сессии.

### 6.5 Аутентификация и авторизация

- Пароли хэшируются через `bcrypt` с автоматической генерацией соли.
- При успешном входе выдаётся JWT-токен (подписан секретным ключом, срок действия — 7 дней).
- Токен хранится в localStorage фронтенда и передаётся в заголовке `Authorization: Bearer <token>` каждого запроса.
- Middleware FastAPI проверяет токен и декодирует пользователя для каждого защищённого эндпоинта.
- Роль `is_admin` требуется для управления пользователями и загрузки данных.

### 6.6 Развёртывание

Система запускается одной командой `docker compose up --build`:

```yaml
services:
  postgres:  # PostgreSQL 16-alpine, порт 5432
  backend:   # Python 3.12 + FastAPI, порт 8000, depends_on: postgres
  frontend:  # Node.js 20-alpine + Next.js 15, порт 3000, depends_on: backend
```

При первом запуске Alembic автоматически применяет миграции и создаёт схему базы данных.

### 7.2 Тестирование скоринг-алгоритма

Проверка корректности весовой формулы:
- При равных значениях трафика, инфраструктуры и транспорта — район с дорогой арендой (Medeu: 12 000 тг/м²) получает score на 12–15 единиц ниже, чем район с дешёвой арендой (Nauryzbay: 4 615 тг/м²).
- Наличие 3+ конкурирующих коворкингов в радиусе 700 м снижает `competition_scaled` до 0.8+, что даёт штраф −0.12 к итоговому баллу.
- Ячейки с нулевой активностью получают `avg_density_scaled = 0` и попадают в нижнюю часть рейтинга.

### 7.3 Тестирование пользовательского интерфейса

**Карта:** загрузка тепловой карты проверялась на партии телеком-данных с 2 847 ячейками. Время загрузки — менее 3 секунд.

**Сравнение:** проверена корректность отображения колонки «Аренда тг/м²» для всех трёх источников данных (кандидаты модели, сессия карты, сохранённые точки) — значения совпадают с `rent_index.json`.

**i18n:** переключение языков проверено для всех трёх языков на всех страницах; отсутствие непереведённых строк подтверждено TypeScript-компилятором.

**Адаптивность:** интерфейс проверен на разрешениях 1920×1080, 1366×768 и 375×812 (мобильный).

### 7.4 Тестирование парсера

Скрипт `krisha_scraper.py` был запущен на реальных страницах Krisha.kz. По итогам собрано 455 объявлений. Результаты проверены вручную: медианные значения по районам соответствуют рыночным ожиданиям (центральные районы — Medeu, Bostandyq — дороже, периферийные — Turksib, Nauryzbay — дешевле).

## 8. ЗАКЛЮЧЕНИЕ

В рамках дипломной работы была спроектирована и реализована веб-платформа CoworkWise для аналитики локаций коворкингов в Алматы.

**Достигнутые результаты:**

1. Разработан и реализован алгоритм многофакторного скоринга локаций, учитывающий пешеходный трафик (вес 0.30), деловую инфраструктуру (0.23), транспортную доступность (0.17), конкуренцию (−0.15) и стоимость аренды (−0.15).

2. Реализован серверный REST API на FastAPI с асинхронным доступом к PostgreSQL. Обработка и хранение телеком-данных обеспечены через табличную схему `telecom_grids` / `telecom_stats`.

3. Создан интерактивный клиентский интерфейс на Next.js 15 с тепловой картой трафика, инструментами сравнения и прогнозирования, поддерживающий три языка (русский, английский, казахский).

4. Интегрированы данные из трёх открытых источников: телеком-оператор (трафик), OpenStreetMap (инфраструктура), Krisha.kz (455 объявлений об аренде по 8 районам).

5. Вся система контейнеризирована в Docker Compose и разворачивается одной командой.

**Направления дальнейшего развития:**

- Переход от экспертных весов к обученной модели (логистическая регрессия, градиентный бустинг) — при накоплении 50+ реальных кейсов открытия коворкингов с данными об успехе/закрытии. Архитектура ML-пайплайна для этого уже готова.
- Добавление кластеризации районов (K-Means по трафику и инфраструктуре) без необходимости в размеченных данных.
- Учёт сезонности при наличии данных за 2+ года.
- Мобильное приложение на React Native с повторным использованием существующего API.
